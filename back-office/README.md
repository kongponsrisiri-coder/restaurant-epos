# SiamEPOS Back Office (SEPOS-041)

Internal ops dashboard for managing all SiamEPOS restaurant clients.
Separate Railway + Netlify deployment from the restaurant EPOS ŌĆö not
visible to restaurant clients.

```
back-office/
  server/   Node + Express + Postgres backend  ŌåÆ Railway
  client/   React + Vite frontend              ŌåÆ Netlify
```

## Quick start (local)

### 1. Backend

```bash
cd back-office/server
npm install
export DATABASE_URL=postgres://...        # Railway-style URL
export PGSSL=disable                       # for local pg without TLS
export JWT_SECRET=$(openssl rand -hex 32)  # any 32-byte hex
export OPS_BOOTSTRAP_EMAIL=you@siamepos.co.uk
export OPS_BOOTSTRAP_PASSWORD=letmein123
export OPS_BOOTSTRAP_NAME='Krit'
npm start                                  # listens on :3002
```

On first boot the schema migrations run idempotently and an initial admin
user is seeded from `OPS_BOOTSTRAP_*` (skipped if any `team_users` rows
already exist ŌĆö so safe to leave the env vars set in Railway).

### 2. Frontend

```bash
cd back-office/client
npm install
# Default backend URL is http://localhost:3002 ŌĆö override for staging:
echo 'VITE_OPS_API=http://localhost:3002' > .env
npm run dev                                # opens http://localhost:5174
```

Log in with the bootstrap admin email + password.

## Railway deploy

Two separate services on Railway (or two `railway.json` configs in one
project).

**Backend service**

- Root directory: `back-office/server`
- Build:   `npm install`
- Start:   `npm start`
- Required env vars:
  - `DATABASE_URL` ŌĆö autogenerated when you attach a Postgres plugin
  - `JWT_SECRET` ŌĆö long random hex string
  - `OPS_BOOTSTRAP_EMAIL`, `OPS_BOOTSTRAP_PASSWORD`, `OPS_BOOTSTRAP_NAME`
    (only used on first boot)
- Optional:
  - `HEALTH_TIMEOUT_MS` (default 8000)
  - `HEALTH_KEEP_ROWS`  (default 2000 per client)

Custom domain in Railway: `ops-api.siamepos.co.uk`

**Frontend (Netlify)**

- Site ŌåÆ Build & deploy ŌåÆ Continuous deployment ŌåÆ link this repo,
  base directory `back-office/client`, build command `npm run build`,
  publish directory `back-office/client/dist`.
- Site ŌåÆ Build & deploy ŌåÆ Environment ŌåÆ add `VITE_OPS_API` =
  `https://ops-api.siamepos.co.uk`.
- Custom domain: `ops.siamepos.co.uk`.

If you don't set `VITE_OPS_API`, the client tries to derive the API URL
by replacing `ops.` with `ops-api.` in `window.location.host`, which
works fine when both domains follow the naming convention above.

## How the health check works

Every 5 minutes the backend cron loops through every client with
`status IN ('active','trial')` and GETs `{railway_url}/api/health` on
each one. The restaurant's server (`src/server.js` in the main repo)
exposes that public endpoint and returns:

```json
{ "status": "ok", "orders_today": 42, "last_order_at": "..." }
```

The result + response time get written into `health_checks`, the table
self-prunes to 2000 rows per client, and the dashboard reads the latest
row alongside each client when listing.

A manual ping is available from the client detail page via
`POST /api/health/run` `{ client_id }`.

## Auth

Stateless JWT (signed with `JWT_SECRET`, 7-day expiry). Frontend stores
the token in `localStorage.ops_token` and sends it as
`Authorization: Bearer ŌĆ”`. Three roles:

- `admin`   ŌĆö can do everything, including team management + note deletes
- `support` ŌĆö can read clients, add notes, run health checks, edit
              client subscription fields
- `viewer`  ŌĆö same as support today (separate role kept for the future)

Add new team members from the **Team** page (admin only). Or insert
directly into Postgres with `bcrypt` if you ever lock yourself out.
